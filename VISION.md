# 精准匹配 · 愿景与路线图

> 每个人的数字化——从多条件筛选到智能匹配。

---

## 我们在做什么

找人的本质是**筛选**。

你想找一个身高170+、本科以上、不吸烟、在本地工作的人——这和一个 Excel 表格的多条件筛选没有区别。

精准匹配做的事：**把每个人变成一组可检索的数据，让找人的人用条件筛出目标**。

不是社交软件，没有站内聊天。就是一个大型的、结构化的、可筛选的人的数据库。

目前已有：
- 603 用户数据
- 40+ 维度字段（身高、体重、肤色、脸型、眼型、嘴型、身材、星座、血型、学历、职业、收入、住房、购车、性格、吸烟、饮酒、宗教、宠物、兴趣爱好、标签……）
- 完整的多条件筛选功能
- 联系方式 AES-256-CBC 加密存储
- 管理后台（审核、统计、日志）

## 两阶段演进

### Phase 1：多条件筛选（已完成 ✅）

本质就是**线上 Excel 表格**。

用户发布资料（贴标签），搜索者按条件筛选（AND 逻辑），找到满足全部条件的人，付费解锁联系方式。

已实现功能：
- 用户注册/登录（手机号 + 密码）
- 40+ 字段资料发布
- 关键词搜索 + 多条件 AND 筛选
- 评价体系（用户对用户评价，计算平均分）
- 举报系统 + 管理后台审核
- 保证金机制（上传支付凭证，管理员审核）
- 联系方式加密解锁（AES-256-CBC，付费后管理员解密）
- 滑动验证码（人机验证）
- 管理后台（用户管理、审核、统计、操作日志）

### Phase 2：智能向量匹配（建设中 🚧）

从"人找条件"到"系统找人"。

多条件筛选的问题是：用户必须明确知道自己要什么。但很多时候，"合适"是一种模糊的感觉，不是一组精确的条件。

Phase 2 要做的是：
- 将每个人的多维数据映射为**向量**（embedding）
- 基于向量相似度，自动推荐"最匹配的两个人"
- 两种模式并存：精确筛选（Phase 1）+ 智能推荐（Phase 2）
- 用户可以调节权重（比如"我更看重学历，不太在意身高"）

技术选型（已确定）：
- **向量数据库**：pgvector（PostgreSQL 扩展，与主库共用，部署简单）
- **Embedding 方案**：支持 OpenAI / 兼容 API / Mock 三种模式
- **相似度检索**：HNSW 索引 + cosine similarity
- **匹配算法**：双向匹配（用户资料 ↔ 择偶要求），综合打分
- **可选**：引入 LLM 对匹配结果给出自然语言解释

已完成工作：
- ✅ PostgreSQL 数据库迁移（从 SQLite，数据目录：项目/data/pg_data/）
- ✅ pgvector 扩展编译安装（v0.8.2）
- ✅ 向量表结构设计（user_vectors 表，HNSW 索引）
- ✅ 向量数据库操作模块（db/vector.js）
- ✅ 向量搜索服务（services/vector.service.js）
- ✅ API 路由（routes/vector.js）

待完成：
- ⏳ Embedding API 接入与调优
- ⏳ 批量向量化现有用户数据
- ⏳ 前端智能推荐页面
- ⏳ 匹配结果可解释性

## 设计原则

- **不是社交软件**：没有站内聊天、没有动态流。就是结构化数据 + 筛选。
- **联系方式加密**：数据库里不存明文联系方式，付费后管理员解密。
- **最小安全原则**：够用就好，不过度设计。
- **数据隐私**：不存身份证等敏感数据，加密存储联系方式。

## 核心洞察

> 每个人都是一串数字。找人的本质是在这串数字里做搜索。

从 Excel 式的精确筛选，到基于向量的模糊匹配——这是从"人找条件"到"系统找人"的进化。

Phase 1 解决"我知道我要什么"的问题，Phase 2 解决"我觉得合适但说不清标准"的问题。
